# DRAGON Enterprise Roadmap

## Зачем это нужно

Чтобы проект стоил дороже, он должен перестать быть просто "набором красивых страниц с логикой" и стать системой управления рестораном.

Самые дорогие свойства такого продукта:

- безопасность и управляемый доступ;
- единая backoffice-панель;
- аналитика, которая влияет на прибыль;
- склад, себестоимость и маржа;
- production-инфраструктура без страха "всё упадёт";
- аккуратный и уверенный интерфейс.

## Текущее состояние

Сейчас у проекта уже сильная прикладная логика:

- меню и клиентская витрина;
- касса;
- kitchen/monitor страницы;
- облачный bridge через Python + Supabase;
- зачатки локальной и облачной auth-модели;
- отдельный PO-модуль.

Но для "дорогого" продукта не хватает системного слоя:

- нет единой модели пользователей и ролей;
- нет общей админки;
- нет финансового контура "маржа / расходы / прибыль";
- нет складского слоя и рецептур;
- нет production baseline уровня VPS + HTTPS + backup + monitoring;
- UX не собран в цельный backoffice.

## Приоритетный порядок

### 1. Security first

Это не "техническая гигиена", а обязательный маркер дорогого продукта.

Нужно сделать:

- убрать риск утечки ключей и токенов;
- хранить реальные секреты только локально и на сервере;
- сделать единый login;
- ввести роли `admin`, `cashier`, `kitchen`, `viewer`;
- добавить audit log на критические действия;
- убрать доступ к внутренним API без прав;
- перевести production на secure cookies + HTTPS.

Почему это повышает цену:

- клиент понимает, что данные сотрудников, выручка и платежи защищены;
- можно продавать нескольким точкам и разным сотрудникам;
- уменьшается страх "любой кто знает ссылку может всё открыть".

## Фаза 1. Unified Admin Panel

Нужна одна backoffice-панель вместо набора отдельных технических страниц.

Модули:

- меню;
- категории;
- цены;
- фото;
- модификаторы;
- стоп-лист;
- скидки;
- пользователи и роли;
- смены и настройки точки.

Что важно в интерфейсе:

- единая навигация;
- понятная структура разделов;
- быстрый поиск;
- inline editing там, где это безопасно;
- история изменений;
- мобильная пригодность для планшета и ноутбука.

Почему это повышает цену:

- проект начинает выглядеть как продукт, а не как внутренний прототип;
- владельцу не нужно помнить "какая страница за что отвечает";
- onboarding нового сотрудника становится дешевле.

## Фаза 2. Reports That Matter

Отчёты должны не просто показывать цифры, а отвечать на вопрос "что мне делать завтра".

Обязательные отчёты:

- выручка за день;
- наличка / карта / онлайн;
- популярные блюда;
- средний чек;
- скидки по дням и сотрудникам;
- расходы;
- прибыль;
- маржа по блюдам;
- продажи по часам;
- продажи по категориям.

Нужны фильтры:

- сегодня / вчера / неделя / месяц;
- точка / касса / сотрудник;
- тип заказа: зал / самовывоз / доставка.

Почему это повышает цену:

- владелец платит не за "табличку", а за ответы на вопросы бизнеса;
- это уже ближе к ERP/POS, чем к сайту заказов.

## Фаза 3. Inventory + Food Cost

Это один из самых дорогих модулей, потому что он напрямую влияет на деньги.

Нужно добавить:

- ингредиенты;
- поставщиков;
- закупочные цены;
- рецептуры;
- расход по ингредиентам на блюдо;
- приход / списание / инвентаризацию;
- остатки;
- food cost;
- маржу на блюдо;
- оповещение о низких остатках.

Пример ценности:

- `Phở = 170 грн`
- `себестоимость = 68 грн`
- `маржа = 102 грн`

После этого система уже помогает:

- менять цену не на глаз;
- отключать убыточные блюда;
- видеть, где ест маржу кухня, закупка или скидки;
- планировать закупку по фактическому спросу.

Почему это повышает цену:

- это редкий и очень понятный владельцу функционал;
- он напрямую связан с прибылью, а не только с приёмом заказа.

## Фаза 4. Production Grade Delivery

Чтобы продукт стоил дороже, он должен продаваться как стабильный сервис, а не как "открой ноутбук и держи вкладку".

Нужно сделать:

- VPS;
- домен;
- HTTPS;
- `systemd` сервисы;
- reverse proxy (`Caddy` или `Nginx`);
- backup базы и файлов;
- мониторинг и алерты;
- централизованные логи;
- staging / production разделение;
- инструкция restore после сбоя.

Минимальный production stack:

- Ubuntu VPS
- `systemd` для Python backend
- `Caddy` для HTTPS by default
- Supabase/Postgres для данных
- cron/systemd timers для backup
- Uptime Kuma или Better Stack для мониторинга

Почему это повышает цену:

- клиент покупает надёжность;
- можно уверенно ставить в рабочее место, а не только показывать demo.

## Фаза 5. Premium Interface

Сейчас ценность в логике. Но коммерческая цена растёт быстрее, когда логика упакована красиво.

Что нужно улучшить:

- единый visual language для всех ролей;
- сильная типографика;
- меньше "служебных" экранов, больше product polish;
- dashboard cards, charts, status chips;
- состояния loading / empty / error;
- аккуратные модалки и таблицы;
- быстрые формы редактирования;
- анимации только там, где они помогают.

Принцип:

- клиент должен открыть экран и подумать: "это готовый продукт".

## Что делать первым

Самый выгодный стартовый спринт:

### Sprint A: Security + Foundation

- единая auth-модель;
- роли `admin/cashier/kitchen/viewer`;
- запрет insecure remote mode в production;
- secure env policy;
- audit events;
- базовая admin shell.

Это самый правильный первый шаг, потому что:

- без него опасно развивать отчёты и склад;
- без него админка останется красивой, но хрупкой;
- именно он переводит проект из hobby/internal в sellable/business.

### Sprint B: Unified Admin

- categories
- menu items
- prices
- photos
- stop-list
- discounts

### Sprint C: Finance and Reports

- revenue
- payment split
- average order value
- top dishes
- expenses
- profit

### Sprint D: Inventory and Cost

- ingredients
- recipes
- stock
- write-offs
- food cost
- margin

## Как это продавать

Проект станет дороже не потому, что "кода больше", а потому что появятся 4 продаваемых обещания:

1. "Система безопасна и управляет доступом сотрудников."
2. "В одном месте можно управлять меню, ценами, стоп-листом и скидками."
3. "Система показывает прибыль, а не только заказы."
4. "Есть склад и себестоимость, значит можно управлять маржой."

## Рекомендуемый следующий шаг в коде

Если двигаться правильно, следующий большой implementation slice должен быть таким:

1. Вынести auth и role checks в единый backend слой.
2. Сделать `users/roles/sessions/audit_events`.
3. Перевести кассу, kitchen и admin на единый login.
4. После этого строить unified admin panel.

Именно этот шаг даст лучший фундамент для всех остальных дорогих модулей.
